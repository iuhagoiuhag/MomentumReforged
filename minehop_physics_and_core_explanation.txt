MINEHOP FABRIC MOD — PHYSICS & CORE EXPLANATION
=================================================

This document explains how the Minehop Fabric mod (MC 1.20.4) works internally.
Use this as reference when building your own similar mod.

================================================================================
1. OVERALL ARCHITECTURE
================================================================================

The mod is split into two source sets via Fabric Loom's split-source feature:
  - src/main/java/  → Server/shared code (runs on dedicated server and integrated server)
  - src/client/java/ → Client-only code (rendering, HUD, keybinds)

Entry points:
  - Minehop.java (ModInitializer) — registers everything server-side
  - MinehopClient.java (ClientModInitializer) — registers client-side features

Global state is stored as static fields on the Minehop class (server) and
MinehopClient class (client). This includes:
  - mapList, recordList, personalRecordList, replayList (persistent data)
  - timerManager (tracks active map runs per player)
  - efficiencyMap, gaugeListMap, efficiencyListMap, efficiencyUpdateMap
  - groundedList (tracks players on ground in the current frame)
  - playerMapLocation (maps players to their current zone)

The player_count field expires every 100 ticks and is recalculated by counting
non-creative players per map (ConfigWrapper:END_SERVER_TICK, every 100 ticks).

================================================================================
2. MOVEMENT PHYSICS — THE CORE (LivingEntityMixin.java)
================================================================================

The mod replaces vanilla movement by injecting into LivingEntity.travel() (HEAD,
cancellable=true) and LivingEntity.jump() (HEAD, cancellable=true). It only
affects PLAYER entities.

Config values (defaults):
  sv_friction        = 0.35
  sv_accelerate      = 0.1
  sv_airaccelerate   = 1.0E99 (effectively infinite)
  sv_maxairspeed     = 0.02325
  speed_mul          = 2.2
  sv_gravity         = 0.066
  speed_coefficient  = 1.0

2A. FRICTION (ground only)

The mod uses Source engine-style friction. It only applies when the player was
on ground last tick AND is on ground this tick (fullGrounded = wasOnGround &&
isOnGround()). This means there is NO friction on the first frame after landing,
which is critical for bunnyhopping.

The friction formula:
  friction_value = 1 - (slipperiness * slipperiness)
  drop = speed * sv_friction * friction_value
  new_speed = max(speed - drop, 0)

The velocity is scaled by new_speed/speed:
  velocity *= (new_speed / speed)

The groundedList tracks which players are fullGrounded this tick (used by the
StartEntity to only trigger timer start while grounded).

2B. INPUT -> DIRECTION CONVERSION (MovementUtil.java)

movementInputToVelocity converts strafe (sI/x) + forward (fI/z) input + yaw
into a direction vector. This is standard angle -> vector math:
  x = input.x * cos(yaw) - input.z * sin(yaw)
  z = input.z * cos(yaw) + input.x * sin(yaw)

Input values are normalized if length > 1.0.

The input is first divided by 0.98F to reverse vanilla's internal scaling.

2C. AIR TURNING PENALTY

In the air (not on ground, not climbing), the input (sI, fI) is scaled by the
absolute yaw difference between headYaw and prevHeadYaw:
  sI *= |headYaw - prevHeadYaw|
  fI *= |headYaw - prevHeadYaw|

This means you must turn your mouse to turn in the air — holding strafe keys
alone without yaw change gives zero air acceleration.

2D. ACCELERATION

The mod computes:
  projVel = current_horizontal_velocity · moveDir  (dot product of current
            velocity projected onto the desired movement direction)
  accelVel = sv_accelerate (ground) or sv_airaccelerate (air)

Max velocity on ground:
  maxVel = movementSpeed * speed_mul
  (movementSpeed is the player's base speed attribute, ~0.1)

Max velocity in air:
  base_maxVel = sv_maxairspeed * ((vel * speed_coeff) / vel) = sv_maxairspeed
                (the coefficient cancels out in the current code)
  angleBetween = acos(current_velocity_normalized · moveDir_normalized)
  maxVel *= (angleBetween^3)

This angle^3 scaling means strafing perpendicular to your velocity gives the
highest speed gain. This is the core Source-engine air-strafe mechanic.

If projVel + accelVel exceeds maxVel, accelVel is clamped:
  accelVel = maxVel - projVel

The acceleration vector is:
  accelDir = moveDir * max(accelVel, 0)

New velocity = current_velocity + accelDir

2E. SPEED CAP

If !fullGrounded and horizontal speed exceeds speedCap, the horizontal velocity
is scaled down proportionally. The speedCap is either the config value (default
1000000 — effectively no cap) or a per-player cap set by HNS maps (0.6).

2F. JUMP (LivingEntityMixin.jump())

The jump method is completely overridden:
  yVel = getJumpVelocity()  (vanilla, ~0.42)
  if JUMP_BOOST: yVel += 0.1 * (amplifier + 1)
  setVelocity(current.x, yVel, current.z)

Jump cooldown is set to 0 every tick (in travel()), so there is no delay
between jumps (essential for bunnyhopping).

2G. GRAVITY

Only applied when not in water, not on ladder, not levitating, not slow falling:
  yVel -= sv_gravity  (default 0.066, about 30% of vanilla's 0.08)

Slow falling overrides: gravity = 0.01
Levitating: uses vanilla formula

2H. LADDER PHYSICS

Climbing speed is clamped to ±0.35 horizontally.
Jumping off ladders applies 0.25 velocity in the ladder-away direction.
Ladder movement direction is remapped based on yaw relative to ladder facing.

The ladder release time prevents re-stick for 4 ticks.

2I. CLIMBING SPEED APPLICATION

applyClimbingSpeed() handles both climbing and jumping-off-ladder states.
When climbing (not jumping): horizontal velocity is clamped to [-0.35, 0.35]
  in the axis perpendicular to the ladder facing; y velocity = ladderfI (clamp
  to [-0.35, 0.4]). If no input, y = 0 (stops sliding).
When jumping: x/z are clamped to allow movement off the ladder.

================================================================================
3. OPTIMAL STRAFE ANGLE CALCULATION (findOptimalStrafeAngle)
================================================================================

This brute-force search tests angles from prevYaw-45 to prevYaw+45 in 1-degree
increments (90 iterations). For each angle, it simulates the acceleration
formula and picks the angle that would give the highest resulting speed.

Used to compute gauge deviation. The autostrafer (commented out) would
automatically set the player's yaw to this optimal angle.

================================================================================
4. EFFICIENCY & GAUGE SYSTEM
================================================================================

4A. STRAFE EFFICIENCY (per tick, calculated during acceleration)

  v = sqrt(vx^2 + vz^2)       → new speed after acceleration
  nogainv = sqrt(old_vx^2 + old_vz^2)  → speed before acceleration
  maxgainv = sqrt(nogainv^2 + maxVel^2)  → theoretical max if accelerating at
                                            perfect angle

  efficiency = clamp(((v - nogainv) / (maxgainv - nogainv)) * 100, 0, 100)

This is the percentage of potential speed gain actually achieved.

Values are stored per-tick in efficiencyMap (current tick) and accumulated in
efficiencyListMap. When the player lands, the average efficiency across all air
ticks since last landing is calculated and stored in efficiencyUpdateMap.

4B. GAUGE

  gaugeValue = (flip sign based on sI/fI) * (normalYaw - optimalAngle) * 2

This measures how many degrees off the optimal angle you are, doubled. Negative
means turning one direction, positive the other.

The last 4 gauge values are averaged every 4 ticks for display.

4C. HUD DISPLAY

Speed is calculated client-side as:
  speed = sqrt((x - prevX)^2 + (z - prevZ)^2)  (per frame)
  blocks/sec = speed / 0.05

SSJ displays: "last_jump_speed (jump_count)"
Efficiency displays as percentage with color coding:
  <50%  DARK_RED
  <70%  RED
  <80%  YELLOW
  <90%  GREEN
  <95%  AQUA
  >=95% WHITE

Gauge displays as a visual bar (<---+++--->) horizontal or vertical, with
10-degree increments per tick mark, max 4 ticks each direction. Color coded
by absolute gauge value.

Prespeed shows speed on the frame the player leaves the ground (first air tick
after being grounded).

================================================================================
5. ZONE SYSTEM
================================================================================

All zone entities extend Zone.java which extends MobEntity. They share:
  - No gravity (hasNoGravity=true)
  - No pushing (isPushable=false, doesNotCollide=true, onPlayerCollision empty)
  - No collision (collidesWith=false)
  - Persistent (cannotDespawn=true, isPersistent=true)
  - Only killable via GENERIC_KILL damage type
  - Store a paired_map name string

5A. START ENTITY (StartEntity)

Tick behavior (every 2 ticks):
  - Teleports to center of corner1/corner2 bounding box
  - Sends zone data to all players via PacketHandler.updateZone()
  - Checks if any grounded (!creative, !spectator, in groundedList) player is
    inside the bounding box → starts timer (stores System.nanoTime) and removes
    any existing replay, sets playerMapLocation

5B. END ENTITY (EndEntity)

Server-side: purely visual. Teleports to center, sends zone data.
Client-side (EndRenderer): predicts next position using velocity:
  nextPosition = currentPos + velocity * partialTicks
If nextPosition is inside the end zone bounding box AND timer is running:
  → sends MAP_FINISH packet with map name and elapsed time
  → resets startTime to 0

This is CLIENT-AUTHORITATIVE finish detection.

5C. RESET ENTITY (ResetEntity)

Tick behavior:
  - Sends zone data with check_index
  - Checks if any non-creative, non-spectator player is inside the bounding box:
    → Teleports to map spawn or checkpoint[index-1]
    → Removes timer if going to spawn (check_index=0)
    → Clears inventory
    → Sets playerMapLocation to the StartEntity for that map

5D. GAMEMODE ENTITY (GamemodeEntity)

Ticks only to check if the paired map still exists. If not, kills itself.
Used to track which map a player is on.

================================================================================
6. TIMER & RECORD SYSTEM
================================================================================

6A. TIMER START

Server side: StartEntity detects player in zone while grounded →
  timerManager maps player_name → {map_name: System.nanoTime}
Client side: StartRenderer detects player in zone while grounded →
  startTime = System.nanoTime()
  (both start independently; server validates time later)

6B. TIMER DISPLAY

Client sends current time to server every render frame while timer is running
via SEND_TIME packet. Server broadcasts to spectators with PB comparison.

6C. MAP FINISH

Client sends MAP_FINISH packet (map_name, time in seconds).
Server handles it:
  - Validates time is within ping_limit (300ms) of server-side timer
  - Compares to world record → if faster, announces globally, saves new WR
  - Compares to personal best → if faster, announces to player
  - Saves replay from replayEntryMap
  - Sends record to Discord if set up
  - Assigns/removes LuckPerms "record_holder" parent

6D. DATA PERSISTENCE

  Stored in <world>/MineHop_Data/:
    minehop_maps.json      → mapList (MapData: name, spawn coords, checkpoints,
                              arena, hns, difficulty, player_count)
    minehop_pbs.json       → personalRecordList (RecordData: name, map_name, time)
    minehop_records.json   → recordList (same shape)
    minehop_replays.json   → replayList (Replay objects)

Loaded on ServerWorldEvents.LOAD, saved on UNLOAD.

================================================================================
7. REPLAY SYSTEM
================================================================================

7A. RECORDING (ReplayEvents.java)

Every server tick, for each player with an active timer:
  - Reads current SSJEntry (jump_count, last_jump_speed, efficiency) from
    lastEfficiencyMap
  - Captures position (x, y, z), pitch, headYaw, SSJ data
  - Appends ReplayEntry to replayEntryMap[player_name]

7B. PLAYBACK (ReplayEntity.java)

The replay entity iterates through saved ReplayEntry frames:
  - Teleports to each recorded position
  - Sets yaw, headYaw, pitch
  - When spectators are watching via setCameraEntity, they are teleported to
    the replay position and given the replay's SSJ/efficiency data
  - Loops back to start when reaching the end

7C. STORAGE

Replay object contains: map_name, player_name, time, List<ReplayEntry>
Each ReplayEntry: x, y, z, xrot, yrot, jump_count, last_jump_speed, efficiency

================================================================================
8. HNS (HIDE & SEEK) GAME MODE
================================================================================

8A. TAGGING

Managed by HNSManager.java. State tracked via:
  taggedMap: player_name → boolean (is tagged/it)
  mapHasTaggers: map_name → boolean
  mapTimers: map_name → tick when taggers were last present

Tagging triggers:
  - Fall damage >= 20 (and not landing on stairs) → auto-tag
  - Melee attack from a tagged player within 3.5 blocks and not below

Tagged players glow (setGlowing(true)).

8B. ROUND MANAGEMENT

If no tagger exists on an HNS map, a random player is assigned as tagger.
If all players on a map are tagged:
  → Round ends, all teleported to random checkpoints, tags removed
  → "you got tagged :(" or "you survived as a hider!" messages

If no taggers exist for 3600 ticks (3 minutes), all players are reset.
Countdown messages every 1200 ticks (1 minute).

8C. HNS-SPECIFIC BEHAVIOR

  - Speed cap reduced to 0.6
  - Attack sound cancelled (SoundSystemMixin)
  - Fall damage tagging
  - Melee attack range and height check
  - Entities are pushable during HNS (isPushable mixin checks o_hns)

================================================================================
9. CONFIGURATION SYSTEM
================================================================================

Uses Cloth Config + AutoConfig with Jankson serializer.

MinehopConfig.java defines:
  enabled, fall_damage, nulls
  movement { sv_friction, sv_accelerate, sv_airaccelerate, sv_maxairspeed,
             speed_mul, sv_gravity, speed_coefficient }
  jHud { ssjHud, efficiencyHud, speedHud, prespeedHud, gaugeHud }
  (hidden fields: help_command, minehop_motd, client_validation, bot_token,
   record_channel)

ConfigWrapper.java:
  - Loads config on startup
  - Syncs config to all clients every tick via PacketHandler.sendConfigToClient
  - Manages speed caps for HNS maps
  - Clears non-hotbar inventory and gives INSTAGIB_GUN on arena maps

Client-side config override: The server can override client physics settings via
Networking. If override_config is true, the client uses server-provided values.

================================================================================
10. PACKET / NETWORKING SYSTEM
================================================================================

ModMessages.java defines ~19 packet Identifiers.

Server-to-client packets:
  CONFIG_SYNC — sends all movement config values, speed cap, HNS flag
  ZONE_SYNC — sends entityId, corner1, corner2, map name, check_index
  SELF_V_TOGGLE, OTHER_V_TOGGLE, REPLAY_V_TOGGLE — visibility toggles
  SEND_EFFICIENCY — sends efficiency to specific player
  SEND_SPECTATORS — lists who is spectating you
  CLIENT_SPEC_EFFICIENCY — sends SSJ data to spectators
  OPEN_MAP_SCREEN — opens map selection GUI
  SEND_MAPS — sends all map data
  SEND_RECORDS — sends world records by map
  SEND_PERSONAL_RECORDS — sends player's PBs
  UPDATE_POWER — sends booster block power values
  SET_PLAYER_CHEATER — marks player as cheater (alternate texture)

Client-to-server packets:
  SEND_TIME — current timer value for spectator display
  MAP_FINISH — map name + time when completing
  SERVER_SPEC_EFFICIENCY — SSJ data during run
  HANDSHAKE_ID — mod version validation
  ANTI_CHEAT_CHECK — anti-cheat response

================================================================================
11. ANTI-CHEAT / PLAYER VALIDATION
================================================================================

HandshakeHandler:
  - When a player joins, the server waits for a HANDSHAKE packet with the
    correct mod version
  - If not received within 60 ticks, the player is disconnected
  - Stores validated players in a set to avoid re-processing

ServerPlayNetworkHandlerMixin:
  - Raises vanilla speed limits (100.0F → Float.MAX_VALUE for player movement,
    300.0F → Float.MAX_VALUE for elytra, 100.0 → Double.MAX_VALUE for vehicles)
  - This prevents the server from kicking players for "flying"

Map finish validation:
  - Server-side timer (System.nanoTime) compared to client-reported time
  - Must be within 300ms tolerance to be accepted

================================================================================
12. BOOST BLOCK SYSTEM
================================================================================

BoostBlock (Block) + BoostBlockEntity (BlockEntity):
  - Stores X, Y, Z power values (persisted to NBT)
  - When player stands on BOOSTER_BLOCK, the velocity is modified:
      vel += (x_power, 0, z_power)
      yVel += y_power
  - 5-tick cooldown between boosts (boostTime tracking)
  - /booster command sets power values

================================================================================
13. MISCELLANEOUS FEATURES
================================================================================

13A. NULL BINDS (KeyBindingMixin)

The client mixin can disable sneak for spectators and implement "null movement"
— tracking left/right key state for advanced movement techniques.

13B. VISIBILITY TOGGLES

Commands: /hide self /hide others /hide replay
  - hideSelf: cancels hand rendering (GameRendererMixin), hides status bars,
    health, hotbar (re-implemented custom), XP bar
  - hideOthers: cancels rendering of other player entities (PlayerEntityRendererMixin)
  - hideReplay: cancels replay entity rendering

13C. SPECTATOR SYSTEM

SpectateCommands.java + PacketHandler:
  - /spec <player> — spectates a player (setCameraEntity, clear inventory)
  - /unspec — returns to normal view
  - spectatorList maps target_name → list of spectator names
  - Watched players' timer, SSJ, and efficiency are forwarded to spectators
  - Spectator HUD is disabled (SpectatorHudMixin)
  - Spectator list is displayed on the HUD

13D. JOIN/LEAVE HANDLING

JoinLeaveManager:
  - On join: teleports to world spawn, sends handshake
  - On disconnect: cleans up all per-player maps (timerManager, efficiencyMap,
    etc.), cancels pending map data

13E. INSTAGIB GUN (Arena mode item)

When a map has arena=true:
  - All non-hotbar inventory slots are cleared
  - Slot 0 is set to INSTAGIB_GUN
  - The gun does raycast damage using the INSTAGIB damage type
  - Hit players are teleported to a random checkpoint

13F. RAMP BLOCKS

A custom block with RAMP_HEIGHT property (0-8). These are purely visual/
decorative blocks; no ramp physics are implemented (the HNS fall-tag check
special-cases StairsBlock for landing).

13G. CUSTOM PLAYER MODEL

PlayerEntityModelMixin overwrites getTexturedModelData() to customize the
player model (adds features like ear, cloak, etc.).

================================================================================
14. KEY EQUATIONS SUMMARY
================================================================================

Horizontal speed (blocks/sec):
  speed = sqrt((x - prevX)^2 + (z - prevZ)^2) / 0.05

Ground friction (per tick):
  drop = speed * sv_friction * (1 - slipperiness^2)
  new_speed = max(speed - drop, 0)

Ground acceleration cap:
  maxVel = movementSpeed * speed_mul

Air acceleration cap:
  maxVel = sv_maxairspeed * angleBetween^3
  where angleBetween = acos(vel_norm · moveDir_norm)

Speed gain per tick:
  projVel = vel · moveDir
  accelVel = min(accel, maxVel - projVel)
  newVel = vel + moveDir * accelVel

Efficiency:
  maxgainv = sqrt(oldSpeed^2 + maxVel^2)
  eff = clamp((newSpeed - oldSpeed) / (maxgainv - oldSpeed) * 100, 0, 100)

Gauge:
  gauge = (yaw - optimalAngle) * 2
